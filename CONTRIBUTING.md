# Contributing to ZGate

Thanks for your interest in contributing to ZGate. This document explains how to set up your environment, the conventions we follow, and how to submit changes.

By participating in this project you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md).

## Ways to Contribute

- Report bugs through GitHub issues using the bug report template.
- Request features through GitHub issues using the feature request template.
- Improve documentation in `docs/` or the README.
- Submit code changes via pull requests.
- Add or improve test coverage.

## Development Setup

### Prerequisites

- **Bun** (latest)
- **Rust** toolchain (latest stable) for building the RTK engine
- **Docker** and Docker Compose for the local stack (PostgreSQL, Redis, Mailpit)

### Steps

```bash
git clone https://github.com/zhafrandzaky/zgate.git
cd zgate

bun install
cp .env.example .env
# Edit .env and set at minimum: JWT_SECRET, JWT_ADMIN_SECRET,
# API_KEY_SECRET, CREDENTIALS_ENCRYPT_KEY, MEMORY_ENCRYPT_KEY

docker compose up -d postgres redis mailpit
bunx prisma migrate dev

cd rtk && cargo build --release && cd ..

bun dev
```

The Next.js app runs on port 3000. The Bun WebSocket sidecar runs on port 3001 (`bun run ws-server/index.ts` in a separate terminal). Mailpit UI is at `http://localhost:8025`.

## Branch Rules

We use GitHub Flow.

- `main` is production only. Never push directly to `main`.
- One task or change per branch, branched from the latest `main`.
- Branch naming:
  - `feat/TASK-XXX-<kebab-name>` for a task
  - `fix/<kebab-description>` for a hotfix
  - `chore/<kebab-description>` for deps, config, and non-feature work
- Merge into `main` only via pull request. Delete the branch after merge.

```bash
git checkout main
git pull origin main
git checkout -b feat/TASK-XXX-<name>
```

## Commit Message Format

```
<type>(<scope>): <description>

[optional body]
```

Types: `feat` `fix` `refactor` `docs` `chore` `test` `perf` `style`.

**Forbidden in commit messages (hard rule):** do not include `Co-authored-by:`, `Signed-off-by:` referring to an AI/tool, `Generated by ...`, `Created with ...`, or any tool name tag such as `[Claude]`, `[Cursor]`, `[Copilot]`, `[Windsurf]`. No trailer or footer that could add an AI agent to the GitHub contributor list is allowed.

## Pull Request Process

1. Ensure your branch is up to date with the latest `main`.
2. Keep one logical change per PR.
3. Run `bun test` (and `cargo test` for `rtk/` changes) and make sure they pass.
4. Fill out the pull request template completely, including the related task (TASK-XXX) and testing checklist.
5. Update documentation in `docs/` or the README if behavior changes.
6. Confirm there are no AI signatures in any commit.
7. A maintainer will review. Address CRITICAL and HIGH review feedback before merge.

## Code Style

- TypeScript strict mode. No `any`.
- Path alias `@/*` maps to the project root.
- Server Components by default; use `"use client"` only when needed.
- All environment variables go through `src/lib/env.ts` (Zod validated).
- Format and lint with Prettier and ESLint before committing.
- Follow the UI rules in `AGENTS.md` section 0: no emoji in UI, use `lucide-react` icons; email templates are text only.

## Reporting Bugs and Requesting Features

- **Bugs:** open an issue with the bug report template. Include reproduction steps, expected vs actual behavior, and environment details.
- **Feature requests:** open an issue with the feature request template. Describe the problem first, then the proposed solution and alternatives.

For security vulnerabilities, do not open a public issue. Follow [SECURITY.md](SECURITY.md) instead.
